## 1. 使用方式

本目录按“总览 -> 规范 -> 任务 -> 补充材料”组织，避免同一内容在多份文档里重复描述。

推荐阅读顺序：

1. 先读本文，明确文档入口、版本基线和开发全流程；
2. 再读 `基于混合渲染与PgVector向量检索的智能内容服务平台 - 企业级开发文档.md`，了解整体架构与业务边界；
3. 进入前后端规范与任务清单，按模块开发；
4. 联调、测试、部署时回到本文执行检查清单。

## 4. 全流程执行手册

### 4.1 环境准备

- 安装 `Python 3.12+`、`Node.js 20+`、`npm 10+`、`Docker`、`Docker Compose`
- 确保本地可用 `PostgreSQL 16 + pgvector` 与 `Redis 7`，或直接使用 Docker
- 复制环境变量模板：
  - `backend/.env.example -> backend/.env`
  - `frontend/.env.example -> frontend/.env`

### 4.2 启动方式

Docker 启动：

```bash
docker compose up -d
docker compose exec backend alembic upgrade head
docker compose exec backend python scripts/seed_db.py
```

本地开发：

```bash
# backend
cd backend
pip install -e ".[dev]"
alembic upgrade head
python scripts/seed_db.py
uvicorn app.main:app --reload

# frontend
cd frontend
npm install
npm run dev
```

### 4.3 日常开发顺序

1. 先确认需求影响范围：页面、接口、模型、RAG、监控、部署。
2. 若改数据库模型，先更新 SQLAlchemy 模型与 Alembic 迁移。
3. 若改接口契约，更新后端 `schemas`、OpenAPI，并同步前端 `types/api.d.ts`。
4. 若改页面交互，优先复用现有组件与样式 token，避免重复造轮子。
5. 若改 RAG 链路，至少检查清洗、切片、检索、生成四段是否仍然闭环。
6. 改完先跑模块测试，再跑类型检查、关键链路验证，最后再提交。

### 4.4 联调顺序

推荐按以下顺序联调，减少排查成本：

1. `数据库 / Redis` 正常；
2. `backend` 启动成功，`/docs` 与 `/openapi.json` 可访问；
3. 执行数据库迁移与种子数据；
4. `frontend` 启动成功，首页、文章列表、后台登录页可打开；
5. 验证后台登录、文章 CRUD、RAG 问答、监控上报四条主链路；
6. 最后验证 Docker 场景与生产环境变量。

### 4.5 测试与质量门禁

建议最少执行以下命令：

```bash
# backend
cd backend
pytest tests/ -v

# frontend
cd frontend
npm run test
npm run typecheck
npx playwright test
```

质量门禁：

- 不新增明显重复文档；
- 不保留过期命令、失效路径、错误截图引用；
- 文档中的端口、服务名、环境变量名必须与代码一致；
- 状态类文档必须标注日期和适用范围，避免误导。

### 4.6 发布与验收

发布前检查：

1. 环境变量齐全，默认密钥已替换；
2. `docker compose up -d --build` 可成功启动；
3. 数据迁移可执行，种子数据可选执行；
4. 首页、文章详情、问答页、后台登录、后台管理页可正常访问；
5. 后端接口文档、前端关键测试、RAG 闭环至少完成一次人工验证；
6. README 与本文中的命令、路径、版本号已同步。
