# 一、前端架构总览

```
┌─ Presentation ──────────────────────────────────┐
│  Layouts → Pages → Components (按业务域拆分)      │
│  styles/: tokens.css, global.css, base.css       │
├─ Logic ─────────────────────────────────────────┤
│  Composables → Pinia Stores → API Client         │
│  (useFetch / $fetch + OpenAPI 自动生成的类型)      │
├─ Infrastructure ────────────────────────────────┤
│  Monitor SDK (Nuxt plugin, 独立 package)          │
│  Router Middleware (鉴权 + PV 采集)                │
│  Error Boundary (<NuxtErrorBoundary>)            │
├─ Build & Deploy ────────────────────────────────┤
│  Nuxt routeRules (SSG / SSR / CSR)              │
│  Docker multi-stage → Node / Nginx / CDN         │
└─────────────────────────────────────────────────┘
```

分层原则：
- Presentation 只负责渲染，不包含业务逻辑
- Logic 层的 Composable 是无状态请求封装，Pinia Store 管理全局持久化状态
- Infrastructure 层通过 Nuxt plugin 注入，与业务代码解耦
- Build & Deploy 由 Nuxt 配置驱动，不侵入业务代码

# 二、产品定位

本项目不做传统个人博客，也不做重型企业后台，而是一个「技术内容站 + 智能知识库 + 前端工程化展示台」。

用户打开网站后，直观看到三件事：
1. 这是一个真实可读的技术内容站
2. 站内内容可以通过 RAG 智能问答检索
3. 项目具备前端工程化深度：混合渲染、性能监控、异常采集、后台管理与可视化

核心关键词：技术写作 · 智能检索 · Nuxt3 混合渲染 · Markdown 阅读体验 · 前端监控 SDK · 内容后台管理 · RAG 引用来源

# 三、联调与环境变量设计

## 3.1 环境变量 (.env)

开发阶段在前端根目录创建 `.env` 文件，生产环境在 Docker 部署时注入：

```bash
# 接口基础地址
NUXT_PUBLIC_API_BASE=http://localhost:8000/api/v1

# 监控上报地址（如果独立部署监控服务）
NUXT_PUBLIC_MONITOR_ENDPOINT=http://localhost:8000/api/v1/monitor/report

# 站点信息
NUXT_PUBLIC_SITE_NAME="智能内容平台"

# SSR 阶段请求后端的内网直连地址（可选，用于生产环境容器间通信加速）
# NUXT_API_INTERNAL_BASE=http://backend:8000/api/v1
```

> 注意：Nuxt3 中 `NUXT_PUBLIC_` 前缀的变量会暴露给客户端。服务端特有的密钥（如 `NUXT_API_INTERNAL_BASE`）不加 `PUBLIC_` 前缀。

## 3.2 接口契约生成 (OpenAPI)

本项目后端 FastAPI 会自动生成 OpenAPI 文档，前端可基于此自动生成 TS 类型契约。

**生成命令示例 (基于 openapi-typescript)：**
```bash
# 从本地运行的后端服务拉取 OpenAPI Schema 并生成 types/api.d.ts
npx openapi-typescript http://localhost:8000/openapi.json -o ./types/api.d.ts
```

**请求封装约定 (composables/useApi.ts)：**
- 使用 `$fetch.create` 封装全局请求实例，注入 `baseURL` 和拦截器。
- 对于 SSR 请求，使用 `useFetch` 包装内部逻辑，确保数据随 HTML 下发，避免水合不一致。

# 四、设计基调

## 4.1 视觉方向

整体采用「安静、克制、技术编辑部」风格，介于技术博客、知识库和轻量工程控制台之间。

避免：
- 大面积紫色渐变
- 玻璃拟态堆叠
- 无意义科技背景图
- 过度营销式首页
- 只有卡片堆叠、没有内容密度的布局

推荐：
- 浅色纸张背景或深色代码编辑器风格
- 清晰的正文排版
- 稳定的网格系统
- 小面积强调色
- 真实文章与真实数据
- 少量精准动效

## 4.2 色彩系统

首版浅色主题，暗色模式列入 P2。

| Token | HEX | 用途 |
|-------|-----|------|
| `--bg-page` | `#FAFAF7` | 页面背景 |
| `--bg-surface` | `#FFFFFF` | 卡片、文章正文区背景 |
| `--text-primary` | `#1F2933` | 主文本 |
| `--text-secondary` | `#667085` | 次级文本（需验证对比度 ≥ 4.5:1） |
| `--text-tertiary` | `#9CA3AF` | 辅助文本、时间戳 |
| `--border-default` | `#E4E0D8` | 卡片边框、分割线 |
| `--accent` | `#0F766E` | 链接、按钮、选中态 |
| `--accent-warm` | `#B7791F` | 警告、图表辅色 |
| `--code-bg` | `#111827` | 代码块背景 |
| `--code-text` | `#E5E7EB` | 代码块文本 |

强调色仅用于链接、按钮、状态标记和图表关键数据，不用于大面积装饰。

## 4.3 字体系统

| 层级 | 字体 | 说明 |
|------|------|------|
| 中文正文 | system-ui, "Noto Sans SC" | 系统默认中文字体，零加载成本 |
| 英文/代码 | JetBrains Mono | 代码块及行内代码，font-display: swap |
| 标题 | "Noto Serif SC", serif（可选） | 衬线标题，如不使用则回退到系统字体 |

字体加载策略：
- 中文不使用 Web Font，避免 CLS
- JetBrains Mono 子集化，仅加载代码块用到的字符
- 所有自定义字体使用 `font-display: swap`

# 七、水合容错策略

Nuxt3 SSR 的核心风险是服务端 HTML 与客户端 VNode 不匹配导致水合失败。具体策略：

**Markdown 渲染**
- 正文在服务端完成基础 Markdown → HTML 转换
- 客户端仅挂载交互增强（代码复制按钮、图片灯箱），不重新渲染正文 DOM
- MarkdownIt 插件在服务端和客户端使用相同版本和配置，确保输出一致

**图表（ECharts）**
- SSR 时渲染占位 `<div>`（固定高度），避免空白区域
- `onMounted` 中初始化 ECharts 实例，不在 setup 阶段执行
- 图表容器添加 `v-once` 或 key 防止水合替换

**第三方脚本**
- 全部通过 `onMounted` 或 Nuxt plugin（client only）加载
- 不使用 `<script>` 标签内联在模板中

**通用兜底**
- Nuxt layout 中使用 `<NuxtErrorBoundary>` 包裹 `<slot />`
- 水合失败时页面仍然可交互（CSR 降级）

# 九、可访问性（a11y）

遵循 WCAG 2.1 AA 最低标准，重点关注：

## 9.1 语义化 HTML

- `<nav>` 导航栏，配合 `aria-label="主导航"`
- `<main>` 主内容区
- `<article>` 文章正文
- `<aside>` 侧边栏（目录）
- `<time>` 发布时间
- `<footer>` 页脚

## 9.2 键盘导航

- 页面顶部 `Skip to content` 链接（第一个可聚焦元素）
- 所有交互元素（链接、按钮、输入框）可通过 Tab 访问
- 自定义组件（下拉菜单、对话框）实现箭头键导航
- RAG 悬浮窗：打开时焦点移至输入框，关闭时焦点回到触发按钮

## 9.3 对比度

- 正文文本 `#1F2933` / `#FAFAF7` = 对比度 14.2:1 ✅ AAA
- 次级文本 `#667085` / `#FAFAF7` = 需实测，接近 4.5:1 边界
- 如 `#667085` 不达标，调整为 `#5A6C7D`
- 链接 `#0F766E` 不以颜色作为唯一区分方式（加下划线）

## 9.4 动效尊重用户设置

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
  }
}
```

# 十、状态管理设计

## 10.1 Pinia Stores（全局持久化状态）

| Store | 数据 | 持久化 |
|-------|------|--------|
| `useAuthStore` | JWT token, 管理员信息, 登录时间 | localStorage（仅后台） |
| `useRAGStore` | 对话历史（最近 N 轮）, 对话 ID | sessionStorage（跨页面保留） |
| `useAppStore` | 主题偏好, 侧边栏折叠状态, 通知 | localStorage |

说明：首版为降低实现成本，后台登录态暂存 localStorage；生产环境建议改为 HttpOnly Cookie，并配合 CSRF 防护与短时 access token，降低 XSS 后 token 泄露风险。

## 10.2 Composables（无共享状态 / 调用级状态）

| Composable | 职责 | 状态范围 |
|-----------|------|---------|
| `useArticles` | 文章列表/详情请求封装 + 响应式数据 | 调用方组件 |
| `useCategories` | 分类列表请求 | 调用方组件 |
| `useTags` | 标签列表请求 | 调用方组件 |
| `useSearch` | 搜索请求 + 结果 | 调用方组件 |
| `useRAG` | 问答请求 + 流式响应处理 | useRAGStore 消费方 |
| `useTracker` | 监控 SDK 初始化 + 生命周期绑定 | Nuxt plugin 内 |
| `useWebVitals` | web-vitals 库的 Vue 封装 | Nuxt plugin 内 |
| `useSeo` | useSeoMeta 便捷封装 | 无状态 |

## 10.3 边界规则

- Composable 可以读取 Pinia Store，但不能写入——写入操作在组件中调用 Store action
- 同一 Composable 被多个组件调用时，每次调用创建独立状态副本
- 需要跨组件/跨路由共享的数据（登录态、对话历史），必须放入 Pinia Store

# 十一、前端监控 SDK 架构

监控 SDK 不是业务功能，是基础设施。作为项目核心前端亮点，独立设计架构。

## 11.1 SDK 分层

```
┌─ Plugin Layer ────────────────────────────────┐
│  Nuxt plugin (client only)                     │
│  注入: $tracker, useTracker()                  │
│  用途: 全局初始化, 生命周期绑定                  │
├─ Collector Layer ─────────────────────────────┤
│  PVCollector        → 路由变化 → PV 事件       │
│  WebVitalsCollector  → web-vitals 库封装       │
│  ErrorCollector     → window.onerror +         │
│                        unhandledrejection      │
│  ExposureCollector  → IntersectionObserver     │
│  ResourceCollector  → 资源加载失败监听          │
│  各 Collector 独立，通过事件总线通信             │
├─ Queue Layer ─────────────────────────────────┤
│  EventQueue: 双阈值刷出（10 条 / 30 秒）        │
│  PriorityChannel: 异常高优独立通道，不参与合并    │
│  Transport: fetch (批量) + sendBeacon (卸载兜底) │
├─ Config Layer ────────────────────────────────┤
│  env, samplingRate, endpoint, debug            │
│  开发环境 100% 采集，生产环境分级采样             │
└───────────────────────────────────────────────┘
```

## 11.2 采集中间层设计

- 各 Collector 实现统一接口：`{ init(), destroy(), onReport(callback) }`
- Collector 之间不直接通信，通过 EventQueue 解耦
- ExposureCollector 通过 `v-track-exposure` 指令绑定 DOM 元素，避免组件包裹地狱

## 11.3 上报分级

| 优先级 | 事件类型 | 通道 | 采样 |
|--------|---------|------|------|
| critical | JS 异常, Promise 异常, 接口 5xx | 高优独立通道 | 100% |
| high | Web Vitals, 接口 4xx, 资源加载失败 | 高优通道 | 100% |
| normal | PV, 停留时长, 元素曝光 | 批量合并 | 生产 10% / 开发 100% |

## 11.4 sendBeacon vs fetch 选择逻辑

```
发送条件：
  if (页面正在卸载) {
    if (数据大小 <= 64KB) → navigator.sendBeacon(url, data)
    else → fetch(url, { method: 'POST', body: data, keepalive: true })
  } else {
    fetch(url, { method: 'POST', body: data })
  }
```

## 11.5 SDK 独立发包

监控 SDK 从业务代码中抽离为独立 npm package（`@project/tracker-sdk`），便于独立维护、版本管理与复用。
